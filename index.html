第二周进度

第二周工作总结 (2025.08.11 ～ 2025.08.17)

1. expected 类迁移与实现

本周我将自己实现的 expected 类成功整合到项目中。该类的核心思想是利用 union 在底层存储两个模板参数对象，并由 expected_storage 类负责管理其构造与析构。

在实现过程中，我遇到并解决了以下问题，并从中收获了宝贵的经验：

  • 空间优化：使用 [[no_unique_address]] 属性，有效压缩了存储空间。
  • 异常安全：确保赋值运算符的异常安全性。
  • 类型优化：利用 类型萃取 (type traits) 机制，对不同模板类型进行优化，例如：
    • 对于 POD 对象，可忽略析构操作。
    • 对于支持无异常移动的对象，优先使用移动语义。
  • 编译期检查：利用 static_assert 提升了编译期报错的友好性。
  • 元编程：通过元编程判断操作是否可能抛出异常。

实践感受：
在处理右值/左值引用、move 与 forward、以及返回值带引用等问题时，为了支持各种类型，整个调试和设计过程极具挑战性。为了简化链式调用，我的实现与标准库略有不同：标准库返回新的 expected 对象，而我返回自身引用。这个设计在多重并发等复杂场景下可能会引发问题，这一点值得深思。

2. 项目结构优化与编译配置

我调整了项目的代码组织结构，使其与 Iceoryx 的多级 CMake 编译配置保持一致。

主要收获：

  • 大型项目组织方式：初步了解了大型项目的模块化组织。例如，hoofs 模块存放与业务无关的代码（如 IPC 封装、expected 等），并编译为静态库；而 posh 模块存放业务相关代码，通过链接 hoofs 中的库来调用其功能。
  • 库头文件管理：学习了库头文件的暴露方式，以及如何通过 interface 间接传递头文件。

待思考：
是否需要将模块编译为动态库？动态库与静态库的利弊是什么？

3. IPC 接口抽象

我抽象出了 ipc_interface_user 和 creator 类，用于控制 IPC channel 的创建与使用。

当前问题：
目前的实现强依赖于 UnixDomainSocket，尚未实现 IPC 接口的泛化。

可选思路：

  • 模板化：使用模板实现不同底层 IPC 机制的切换。
  • 编译期配置：参考 Iceoryx 的做法，在编译期通过 CMake 指定底层实现。

4. 进程注册状态机

Pub 端的状态机流程已基本实现，分为以下四个状态：

  1. 初始化：重新打开端口。
  2. 发送态：向 RouDi 发送注册信息（目前为 “hello world”）。
  3. 接收态：接收 RouDi 的响应（目前为 “ye hello world”）。
  4. 结束态：完成注册流程。

下周计划

  • 实现完整的进程注册流程，确保其稳定可靠。
  • 考虑 IPC 接口的泛化，初步评估两种可选方案的优劣。

第一周进展

本周进展（20250805 - 20250810）

1. 通信链路搭建

完成了 Unix 域套接字（Unix Domain Socket）在 RouDi 和发布者之间的通信链路搭建。

  • 抽象出 UnixDomainSocket 类，并采用 RAII（Resource Acquisition Is Initialization）原则管理套接字生命周期，确保资源自动释放。

2. 构造与错误处理优化

为避免在 UnixDomainSocket 构造函数中直接抛出异常，我引入了 UnixDomainSocketBuilder。

  • 提供了 create() 接口来统一创建对象。
  • 为了解决 create() 返回对象本身时依赖异常机制的问题，我引入了 expected 作为返回值。
    • 优点：避免异常的性能开销，保持错误处理的一致性，并支持链式调用。

3. expected 的实现与学习

通过阅读 LLVM expected 源码，我成功实现了一个简化版本，并从中掌握了以下关键知识点：

  • [[no_unique_address]] 的用途。
  • 如何避免通用模板过度匹配（“万金油构造”）的问题。

4. 异常性能调研

我编写了一个 Demo 来测试异常的性能消耗，并通过分析其汇编代码，深入了解了以下内容：

  • -g 编译选项引入的额外消耗。
  • GCC 异常栈展开及 throw 操作的成本。

  • 博客文章二：撰写一篇关于 expected 的科普博客。一方面解释为什么需要 expected，另一方面从实现角度讲解其基础方法和进阶思路。
