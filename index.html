第五周进度

【2025.09.15 ～ 2025.09.21】 复刻进度总结

当前目标：聚焦·注册模块·的实现，围绕进程注册、内存分块细化、共享内存注册三个方向推进。

1.进程注册
· 实现：
· roudi维护了一个进程集合。（这个不存储在共享内存中）
· pub/sub初始化时会上报进程名，roudi检查是否已存在。
· 若存在，说明旧进程重启了，需要先执行清理操作（目前没有XD，应该还不需要…，但是我象征性的先删除=。=），再加入集合。

2.内存分块细分
· 我的实现：
· pub/sub不直接感知共享内存名
· 注册阶段，roudi通过ipc返回 sgementId -> shmName的映射关系。
Ps：这不是完全解偶吗？
· Iceoryx设计：
· 所有映射关系存储在一个指定的共享内存中，该共享内存由“managment” + 域名拼接而成
· pub/sub注册后，从该共享内存直接读取映射关系。
· FromAI：
·我的方案：
· 优点：解偶 + 简单
· 缺点：
· IPC开销：伴随着进程数量的增加，IPC可能成为性能瓶颈。
· 实时性和一致性：如果roudi重启，所有客户端需要重新获取映射关系，并且通过IPC映射信息，保证即使最新、一致。
· iceoryx设计：
· 优点：（好呀，好呀，只说优点是吧）
· 极致性能：一旦客户端通过IPC注册成功，它可以直接从共享内存中读取所有映射关系，无需再次通信。共享内存的速度远远快于IPC。
· 伸缩性：性能不会随着进程数的增加而降低。
· 状态一致性：天然的状态一致性。 // 这个应该不会体现吧，因为Iceoryx好像不会动态变化吧，roudi重启，会怎么样呢？

3.共享内存注册
该模块主要包含两个核心功能：
（1） SharedMemoryUser类
· 功能：pub/sub通过该类构建共享内存映射关系
· 改进：
· 自己实现的SharedMemoryUser接口和作用与Iceoryx还是又差别的，我觉得一方面是因为源码设计优秀，一方面我认为是我有的功能还没有实现。不过我能理解Iceoryx这么设计的原因。
· 在之前共享内存相关类中增加了AccessMode和OpenMode参数，代码复用性显著提升。
· 实现了简易的ScopeGuard（RAII）
· 功能：用于在函数退出前优雅的执行清理逻辑
· 优点：
· 减少代码冗余
· 避免遗漏关键操作
· 思考：
· Iceoryx的ScopeGuard基于栈分配，这不是唯一一个栈分配，设计很高效。todo. 等我核心流程跑通后，我将研究该细节。
（2） 指针仓库
· 功能：维护 segmentId -> shm 的映射关系，用于统一管理已打开的共享内存，避免重复映射？
· 接口返回值抉择的思考：
· bool：用于简单的，错误分支有限的场景。
· optional：用于”有或无”的返回值（比如返回类的指针成员？）
· expected：适用于有多种错误，并需要返回详细错误的情况。
· 思考：
· 为什么引入 segmentId ？直接通过名字有什么问题吗？应该都行吧。
From AI：
· 解偶，如果底层实现从共享内存切换到其他IPC机制时，segmentId仍可作为统一的标识符，无需修改上层代码。
· 简化：整数ID比处理一个冗长的文件名简单
· 安全：特殊字符、攻击、伪造/冲突。
4.todo
· 在每一个chunk中，引入内存池。
· 设计pub/sub信息存储方案，确保后续匹配与注销操作高效与可扩展。
Ps：虽然整理架构已清晰，但感觉“差的不多，又差很多”😂，争取本月完成注册模块。

第四周进度

【2025.09.08 ~ 2025.09.12】

1.共享内存的创建与封装

· 我的实现：将共享内存的创建与映射逻辑封装到一个类中
· Iceoryx实现：分为 PosixSharedMemory (创建共享内存) 和 PosixMemoryMap (mmap), 再通过 PosixSharedMemoryObject 组合两者
· 对比与思考
· 可重用性更好：如果需要新增 file+mmap 实现，只需要增加一个 PosixFileManager 用于文件创建与管理, 再新增 PosixFileMemory 组合 PosixFileManager 和 PosixMemoryMap 即可。
· 资源管理更安全：如果将整个逻辑放在一个类中，如果 mmap 失败需要手动调用 shm_unlink, 但 Iceoryx 借助 RAII 可以在 PosixMemoryMap 创建时候后自动关闭

2.内存分块及管理

· 实现思路：借鉴 Iceoryx, 抽象出:
· MemoryManager: 统一内存管理类
· MemoryProvider: 内存分配器 (shm, file, …)
· MemoryBlock: 具体内存块 (可对应具体的功能块: 心跳区, 数据区…)
· 思考
· 可扩展性好
· 动态设计合理:
· MemoryManager 内存保存 MemoryProvider * 数组, 不同 MemoryProvider 可以通过多态采用不同的分配策略 (shm+mmap, file+mmap)
· MemoryProvider 内存保存 MemoryBlock * 数组, 也可以扩展为不同特性的 MemoryBlock
· 命名与职责划分思考: 共享内存文件名的组装应该放到 Provider 中,而不是在 PosixSharedMemoryObject/PosixSharedMemory/PosixMemoryMap 中
· MemoryProvider 使用了模板方法模型,将算法的流程和具体实现分离，比如：
· Create 是一个共有接口，这里函数实现了具体内存创建和分块的主要流程。
· CreateMemory 是一个纯虚函数，子类实现具体的内存创建逻辑，可能是 shm+mmap, 可能是 file+mmap…
· Create 中会调用 CreateMemory
// 我感觉这也是希望尽可能保证代码复用吧.

3.expected类优化

· 新增以下功能:
· operator ->
· operator *
· has_error()
· bool 转换
· 下一步计划: 增加 ok/err 包装类可更方便的使用 expected 以及可读性

4.C++新知识点

· 理解了 std::addressof 和 & 的区别:
· & 可以被重载,不一定能获取对象的真实地址.
· std::address_of 始终能获取对象的真实地址.

第三周进度

【2025.08.25 ～ 2025.09.01】复刻项目进度

本周主要完成以下工作：

  1. 优化 pub 端注册的状态机处理循环
    · 实现了发送超时与接收超时逻辑，为通信提供了实时性支持。这里通过套接字选项完成；由于 connect 本身没有超时限制（iceoryx 中也未实现），我推测是因为 Unix 域套接字的连接建立速度很快。
    · 将超时判断逻辑抽象到 deadline_timer 中，避免在状态机中显式获取当前时间并做比较，从而让状态机代码更简洁。
    · 其他小规模调整与优化。

  2. 封装 IpcMessage 类
    用于拼接通信消息，重载了 << 运算符，提升了代码可读性和使用便捷性。

  3. 抽象 IPC 接口层次
    · 新增 IpcInterfaceCreator 与IpcInterfaceUser，分别用于服务端与客户端的创建。
    · 底层抽象出 IpcInterface 模板，支持多种通信方式（例如之前实现UnixDomainSocket）。
    · 借助 IpcInterfaceBase（using IpcInterfaceBase = IpcInterface）作为中间层进行解耦，同时通过宏来灵活切换不同 IPC channel 的具体实现。

在实现过程中学习了 模板类声明与定义分离的两种方法：

  1. 使用 .inl 文件，并在头文件中 #include；
  2. 显式实例化具体类型，例如在头文件中声明 IpcInterface。

第二周进度

第二周工作总结 (2025.08.11 ～ 2025.08.17)

1. expected 类迁移与实现

本周我将自己实现的 expected 类成功整合到项目中。该类的核心思想是利用 union 在底层存储两个模板参数对象，并由 expected_storage 类负责管理其构造与析构。

在实现过程中，我遇到并解决了以下问题，并从中收获了宝贵的经验：

  • 空间优化：使用 [[no_unique_address]] 属性，有效压缩了存储空间。
  • 异常安全：确保赋值运算符的异常安全性。
  • 类型优化：利用 类型萃取 (type traits) 机制，对不同模板类型进行优化，例如：
    • 对于 POD 对象，可忽略析构操作。
    • 对于支持无异常移动的对象，优先使用移动语义。
  • 编译期检查：利用 static_assert 提升了编译期报错的友好性。
  • 元编程：通过元编程判断操作是否可能抛出异常。

实践感受：
在处理右值/左值引用、move 与 forward、以及返回值带引用等问题时，为了支持各种类型，整个调试和设计过程极具挑战性。为了简化链式调用，我的实现与标准库略有不同：标准库返回新的 expected 对象，而我返回自身引用。这个设计在多重并发等复杂场景下可能会引发问题，这一点值得深思。

2. 项目结构优化与编译配置

我调整了项目的代码组织结构，使其与 Iceoryx 的多级 CMake 编译配置保持一致。

主要收获：

  • 大型项目组织方式：初步了解了大型项目的模块化组织。例如，hoofs 模块存放与业务无关的代码（如 IPC 封装、expected 等），并编译为静态库；而 posh 模块存放业务相关代码，通过链接 hoofs 中的库来调用其功能。
  • 库头文件管理：学习了库头文件的暴露方式，以及如何通过 interface 间接传递头文件。

待思考：
是否需要将模块编译为动态库？动态库与静态库的利弊是什么？

3. IPC 接口抽象

我抽象出了 ipc_interface_user 和 creator 类，用于控制 IPC channel 的创建与使用。

当前问题：
目前的实现强依赖于 UnixDomainSocket，尚未实现 IPC 接口的泛化。

可选思路：

  • 模板化：使用模板实现不同底层 IPC 机制的切换。
  • 编译期配置：参考 Iceoryx 的做法，在编译期通过 CMake 指定底层实现。

4. 进程注册状态机

Pub 端的状态机流程已基本实现，分为以下四个状态：

  1. 初始化：重新打开端口。
  2. 发送态：向 RouDi 发送注册信息（目前为 “hello world”）。
  3. 接收态：接收 RouDi 的响应（目前为 “ye hello world”）。
  4. 结束态：完成注册流程。

下周计划

  • 实现完整的进程注册流程，确保其稳定可靠。
  • 考虑 IPC 接口的泛化，初步评估两种可选方案的优劣。

第一周进展

本周进展（20250805 - 20250810）

1. 通信链路搭建

完成了 Unix 域套接字（Unix Domain Socket）在 RouDi 和发布者之间的通信链路搭建。

  • 抽象出 UnixDomainSocket 类，并采用 RAII（Resource Acquisition Is Initialization）原则管理套接字生命周期，确保资源自动释放。

2. 构造与错误处理优化

为避免在 UnixDomainSocket 构造函数中直接抛出异常，我引入了 UnixDomainSocketBuilder。

  • 提供了 create() 接口来统一创建对象。
  • 为了解决 create() 返回对象本身时依赖异常机制的问题，我引入了 expected 作为返回值。
    • 优点：避免异常的性能开销，保持错误处理的一致性，并支持链式调用。

3. expected 的实现与学习

通过阅读 LLVM expected 源码，我成功实现了一个简化版本，并从中掌握了以下关键知识点：

  • [[no_unique_address]] 的用途。
  • 如何避免通用模板过度匹配（“万金油构造”）的问题。

4. 异常性能调研

我编写了一个 Demo 来测试异常的性能消耗，并通过分析其汇编代码，深入了解了以下内容：

  • -g 编译选项引入的额外消耗。
  • GCC 异常栈展开及 throw 操作的成本。

  • 博客文章二：撰写一篇关于 expected 的科普博客。一方面解释为什么需要 expected，另一方面从实现角度讲解其基础方法和进阶思路。
