第五周进度

【2025.09.15 ～ 2025.09.21】Iceoryx 复刻进度总结

当前目标：聚焦 注册模块 的实现，围绕 进程注册、内存分块细化、共享内存注册 三个方向推进。

1️⃣ 进程注册

✅ 实现

  • 进程集合维护
    • roudi 维护一个进程集合（不存储在共享内存中）。
    • pub/sub 初始化时会上报进程名，roudi 检查是否已存在。
  • 重启处理
    • 若存在同名进程，说明旧进程重启：
      • 先执行清理操作（当前版本仅象征性删除）
      • 再加入集合

2️⃣ 内存分块细分

🧩 我的实现

  • pub/sub 不直接感知共享内存名
  • 注册阶段由 roudi 通过 IPC 返回 segmentId -> shmName 的映射关系
  • ✅ 优点：解耦 + 简单

🧊 Iceoryx 设计

  • 所有映射关系存储在一个指定的共享内存中
  • 共享内存命名：management + 域名
  • pub/sub 注册后可直接从共享内存读取映射关系

💡 对比分析

方案 优点 缺点
我的方案 解耦、简单 - IPC 开销随进程数增加可能成为性能瓶颈
- roudi 重启后，客户端需重新获取映射关系
Iceoryx - 极致性能：注册后直接从共享内存读取，速度远快于 IPC
- 伸缩性强：性能不随进程数增加而下降
- 状态一致性天然保证		 - 设计复杂度更高

❓ 备注：Iceoryx 似乎不会频繁动态变化，那 roudi 重启后状态一致性如何保证？后续待验证。

3️⃣ 共享内存注册

该模块包含 SharedMemoryUser 和 指针仓库 两个核心功能。

(1) SharedMemoryUser 类

  • 功能：pub/sub 通过该类构建共享内存映射关系
  • 改进与思考：
    • 自己实现的接口与 Iceoryx 存在差异
      • 部分源于 Iceoryx 的优秀设计
      • 也因我尚未实现全部功能
    • 在共享内存相关类中增加 AccessMode 和 OpenMode 参数 → 复用性提升
    • 实现简易 ScopeGuard（RAII）
      • ✅ 优点：
        • 减少代码冗余
        • 避免遗漏关键清理操作
      • 💡 思考：
        • Iceoryx 的 ScopeGuard 基于栈分配 → 设计高效
        • 待核心流程跑通后深入研究

(2) 指针仓库

  • 功能：维护 segmentId -> shm 的映射关系，统一管理已打开的共享内存，避免重复映射
  • 接口返回值设计：
    返回值 适用场景
    bool 简单、错误分支有限
    optional “有或无”的场景（例如类指针成员）
    expected 错误种类多、需返回详细错误
  • 思考：为什么引入 segmentId，而不是直接使用名字？
    • 解耦：若底层从共享内存切换到其他 IPC 机制，segmentId 仍可作为统一标识符
    • 简化：整数 ID 处理更高效
    • 安全：避免特殊字符、攻击或命名冲突

4️⃣ TODO

  • 在每一个 chunk 中引入内存池。
  • 设计 pub/sub 信息存储方案，确保匹配与注销操作的高效与可扩展。

✍ 小结

架构整理已较为清晰，但依然有“差的不多，又差很多”的感觉。
目标：争取本月完成注册模块 💪

第四周进度

【2025.09.08 ~ 2025.09.12】Iceoryx 复刻进度总结

1️⃣ 共享内存的创建与封装

🧩 我的实现

  • 将 共享内存创建 与 映射逻辑 封装到一个类中
  • 简单直接，但资源管理需手动处理（例如 mmap 失败时需要手动调用 shm_unlink）

🧊 Iceoryx 实现

  • 模块拆分：
    • PosixSharedMemory：负责共享内存创建
    • PosixMemoryMap：负责 mmap 映射
    • PosixSharedMemoryObject：组合上述两个类
  • 优势：
    • 可重用性更高：若新增 file + mmap 实现，只需增加 PosixFileManager（文件创建管理）和 PosixFileMemory（组合 PosixFileManager + PosixMemoryMap）即可
    • 资源管理更安全：借助 RAII，在 PosixMemoryMap 创建失败时可以自动释放资源

💡 对比与思考

方案 优点 缺点
我的实现 简单直接，逻辑集中 失败处理需手动释放资源，扩展性一般
Iceoryx 模块解耦、复用性强，RAII 管理更安全 设计复杂度更高

2️⃣ 内存分块及管理

🧩 实现思路

借鉴 Iceoryx，抽象出三层结构：

  • MemoryManager：统一内存管理类
  • MemoryProvider：内存分配器（shm、file 等）
  • MemoryBlock：具体内存块（可对应功能块，如心跳区、数据区）

💡 思考与设计要点

  • 可扩展性：
    • MemoryManager 内部持有 MemoryProvider* 数组
    • 不同 MemoryProvider 可通过多态采用不同分配策略（如 shm+mmap、file+mmap）
    • MemoryProvider 内部持有 MemoryBlock* 数组，可扩展不同特性的 MemoryBlock
  • 职责划分：
    • 共享内存文件名组装 应放在 Provider 中，而不是 PosixSharedMemoryObject/PosixSharedMemory/PosixMemoryMap
  • 模板方法模式：
    • Create：公共接口，负责内存创建与分块的整体流程
    • CreateMemory：纯虚函数，子类实现具体的内存创建逻辑（shm+mmap、file+mmap 等）
    • 目的：分离流程与实现，提升代码复用性与扩展性

3️⃣ expected 类优化

✅ 新增功能

  • operator->
  • operator*
  • has_error()
  • bool 转换

🔜 下一步

  • 增加 ok/err 包装类，简化 expected 的使用，提高可读性

4️⃣ C++ 新知识点

📌 std::addressof vs &

  • &：可以被重载，不一定能获取对象真实地址
  • std::addressof：始终获取对象真实地址，避免运算符重载带来的歧义

✍ 小结

这一阶段的核心是内存管理架构的清晰化与模块化：

  • 通过模板方法与类分层，保证扩展性和可复用性
  • 通过 RAII 提升资源管理安全性
    同时在 expected 等基础库优化中，进一步夯实 C++ 代码质量。

第三周进度

【2025.08.25 ～ 2025.09.01】Iceoryx 复刻进度总结

1️⃣ 优化 pub 端注册状态机处理循环

✅ 实现内容

  • 超时逻辑
    • 实现发送超时与接收超时，为通信提供实时性支持
    • 通过套接字选项实现（connect 本身无超时限制，Iceoryx 也未实现，推测因 Unix 域套接字连接速度足够快）
  • 代码结构优化
    • 将超时判断抽象到 deadline_timer 中
    • 避免在状态机中显式获取当前时间并做比较
    • 让状态机逻辑更简洁、职责更单一
  • 其他
    • 小规模代码调整与细节优化

2️⃣ 封装 IpcMessage 类

目标

  • 用于拼接通信消息
  • 重载 << 运算符，提升代码可读性与使用便捷性

效果

  • 消息构造更简洁
  • 代码更接近流式表达，减少样板代码

3️⃣ 抽象 IPC 接口层次

结构设计

  • IpcInterfaceCreator
    • 面向服务端的 IPC 创建接口
  • IpcInterfaceUser
    • 面向客户端的 IPC 使用接口
  • IpcInterface 模板
    • 底层抽象，支持多种通信方式（例如 UnixDomainSocket）
  • IpcInterfaceBase
    • using IpcInterfaceBase = IpcInterface<UnixDomainSocket>
    • 作为中间层进行解耦
    • 结合宏可灵活切换不同 IPC channel 的具体实现

设计优势

  • 解耦具体通信方式
  • 支持未来扩展不同的 IPC 通道

4️⃣ 模板类声明与定义分离方法

在实现过程中学习并实践了两种常见方式：

  1. .inl 文件法

    • 将模板类定义放入 .inl 文件
    • 在头文件中 #include 该 .inl 文件

  2. 显式实例化法

    • 在头文件中声明具体类型
    • 例如：IpcInterface<UnixDomainSocket>
    • 在源文件中进行显式实例化

✍ 小结

本周核心成果在于状态机实时性提升与IPC 抽象层完善：

  • 通过 deadline_timer 和 IpcMessage，代码可读性与可维护性显著增强
  • 模板抽象与实例化技巧，为后续扩展多种 IPC 方案奠定了良好基础。

第二周进度

第二周工作总结 (2025.08.11 ～ 2025.08.17)

1. expected 类迁移与实现

本周我将自己实现的 expected 类成功整合到项目中。该类的核心思想是利用 union 在底层存储两个模板参数对象，并由 expected_storage 类负责管理其构造与析构。

在实现过程中，我遇到并解决了以下问题，并从中收获了宝贵的经验：

  • 空间优化：使用 [[no_unique_address]] 属性，有效压缩了存储空间。
  • 异常安全：确保赋值运算符的异常安全性。
  • 类型优化：利用 类型萃取 (type traits) 机制，对不同模板类型进行优化，例如：
    • 对于 POD 对象，可忽略析构操作。
    • 对于支持无异常移动的对象，优先使用移动语义。
  • 编译期检查：利用 static_assert 提升了编译期报错的友好性。
  • 元编程：通过元编程判断操作是否可能抛出异常。

实践感受：
在处理右值/左值引用、move 与 forward、以及返回值带引用等问题时，为了支持各种类型，整个调试和设计过程极具挑战性。为了简化链式调用，我的实现与标准库略有不同：标准库返回新的 expected 对象，而我返回自身引用。这个设计在多重并发等复杂场景下可能会引发问题，这一点值得深思。

2. 项目结构优化与编译配置

我调整了项目的代码组织结构，使其与 Iceoryx 的多级 CMake 编译配置保持一致。

主要收获：

  • 大型项目组织方式：初步了解了大型项目的模块化组织。例如，hoofs 模块存放与业务无关的代码（如 IPC 封装、expected 等），并编译为静态库；而 posh 模块存放业务相关代码，通过链接 hoofs 中的库来调用其功能。
  • 库头文件管理：学习了库头文件的暴露方式，以及如何通过 interface 间接传递头文件。

待思考：
是否需要将模块编译为动态库？动态库与静态库的利弊是什么？

3. IPC 接口抽象

我抽象出了 ipc_interface_user 和 creator 类，用于控制 IPC channel 的创建与使用。

当前问题：
目前的实现强依赖于 UnixDomainSocket，尚未实现 IPC 接口的泛化。

可选思路：

  • 模板化：使用模板实现不同底层 IPC 机制的切换。
  • 编译期配置：参考 Iceoryx 的做法，在编译期通过 CMake 指定底层实现。

4. 进程注册状态机

Pub 端的状态机流程已基本实现，分为以下四个状态：

  1. 初始化：重新打开端口。
  2. 发送态：向 RouDi 发送注册信息（目前为 “hello world”）。
  3. 接收态：接收 RouDi 的响应（目前为 “ye hello world”）。
  4. 结束态：完成注册流程。

下周计划

  • 实现完整的进程注册流程，确保其稳定可靠。
  • 考虑 IPC 接口的泛化，初步评估两种可选方案的优劣。

第一周进展

本周进展（20250805 - 20250810）

1. 通信链路搭建

完成了 Unix 域套接字（Unix Domain Socket）在 RouDi 和发布者之间的通信链路搭建。

  • 抽象出 UnixDomainSocket 类，并采用 RAII（Resource Acquisition Is Initialization）原则管理套接字生命周期，确保资源自动释放。

2. 构造与错误处理优化

为避免在 UnixDomainSocket 构造函数中直接抛出异常，我引入了 UnixDomainSocketBuilder。

  • 提供了 create() 接口来统一创建对象。
  • 为了解决 create() 返回对象本身时依赖异常机制的问题，我引入了 expected 作为返回值。
    • 优点：避免异常的性能开销，保持错误处理的一致性，并支持链式调用。

3. expected 的实现与学习

通过阅读 LLVM expected 源码，我成功实现了一个简化版本，并从中掌握了以下关键知识点：

  • [[no_unique_address]] 的用途。
  • 如何避免通用模板过度匹配（“万金油构造”）的问题。

4. 异常性能调研

我编写了一个 Demo 来测试异常的性能消耗，并通过分析其汇编代码，深入了解了以下内容：

  • -g 编译选项引入的额外消耗。
  • GCC 异常栈展开及 throw 操作的成本。

  • 博客文章二：撰写一篇关于 expected 的科普博客。一方面解释为什么需要 expected，另一方面从实现角度讲解其基础方法和进阶思路。
