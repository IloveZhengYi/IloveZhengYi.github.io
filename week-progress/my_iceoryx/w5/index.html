第五周进度

【2025.09.15 ～ 2025.09.21】Iceoryx 复刻进度总结

当前目标：聚焦 注册模块 的实现，围绕 进程注册、内存分块细化、共享内存注册 三个方向推进。

1️⃣ 进程注册

✅ 实现

  • 进程集合维护
    • roudi 维护一个进程集合（不存储在共享内存中）。
    • pub/sub 初始化时会上报进程名，roudi 检查是否已存在。
  • 重启处理
    • 若存在同名进程，说明旧进程重启：
      • 先执行清理操作（当前版本仅象征性删除）
      • 再加入集合

2️⃣ 内存分块细分

🧩 我的实现

  • pub/sub 不直接感知共享内存名
  • 注册阶段由 roudi 通过 IPC 返回 segmentId -> shmName 的映射关系
  • ✅ 优点：解耦 + 简单

🧊 Iceoryx 设计

  • 所有映射关系存储在一个指定的共享内存中
  • 共享内存命名：management + 域名
  • pub/sub 注册后可直接从共享内存读取映射关系

💡 对比分析

方案 优点 缺点
我的方案 解耦、简单 - IPC 开销随进程数增加可能成为性能瓶颈
- roudi 重启后，客户端需重新获取映射关系
Iceoryx - 极致性能：注册后直接从共享内存读取，速度远快于 IPC
- 伸缩性强：性能不随进程数增加而下降
- 状态一致性天然保证		 - 设计复杂度更高

❓ 备注：Iceoryx 似乎不会频繁动态变化，那 roudi 重启后状态一致性如何保证？后续待验证。

3️⃣ 共享内存注册

该模块包含 SharedMemoryUser 和 指针仓库 两个核心功能。

(1) SharedMemoryUser 类

  • 功能：pub/sub 通过该类构建共享内存映射关系
  • 改进与思考：
    • 自己实现的接口与 Iceoryx 存在差异
      • 部分源于 Iceoryx 的优秀设计
      • 也因我尚未实现全部功能
    • 在共享内存相关类中增加 AccessMode 和 OpenMode 参数 → 复用性提升
    • 实现简易 ScopeGuard（RAII）
      • ✅ 优点：
        • 减少代码冗余
        • 避免遗漏关键清理操作
      • 💡 思考：
        • Iceoryx 的 ScopeGuard 基于栈分配 → 设计高效
        • 待核心流程跑通后深入研究

(2) 指针仓库

  • 功能：维护 segmentId -> shm 的映射关系，统一管理已打开的共享内存，避免重复映射
  • 接口返回值设计：
    返回值 适用场景
    bool 简单、错误分支有限
    optional “有或无”的场景（例如类指针成员）
    expected 错误种类多、需返回详细错误
  • 思考：为什么引入 segmentId，而不是直接使用名字？
    • 解耦：若底层从共享内存切换到其他 IPC 机制，segmentId 仍可作为统一标识符
    • 简化：整数 ID 处理更高效
    • 安全：避免特殊字符、攻击或命名冲突

4️⃣ TODO

  • 在每一个 chunk 中引入内存池。
  • 设计 pub/sub 信息存储方案，确保匹配与注销操作的高效与可扩展。

✍ 小结

架构整理已较为清晰，但依然有“差的不多，又差很多”的感觉。
